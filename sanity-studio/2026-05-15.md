## 1. Media Library — Searchable "By Project" subview

### Question
User asked: searching "Mahogany" in Media Library returned nothing despite media being linked to that project — why?

### Answer
Sanity's global document search indexes only **string fields on the document itself**. The `media.projects` field is an *array of references* — it stores ref pointers (IDs), not the project's title text. The project name appears in the row preview (via `prepare()` dereferencing `projects.0.title`) but never enters the search index.

### Fix
Replaced the single `Media Library` list item in `sanity.config.ts` with a folder containing two children:

```
Media Library
├── All Media           (original full list — for global text search)
└── By Project          → project picker → filtered list via references($projectId)
```

The "By Project" subview filters with `_type == "media" && references($projectId)` so it catches Notices, project-scoped promos. Global-scope promos don't appear here (their `projects[]` is empty) — those still live in "All Media".

**File**: `sanity.config.ts` (Media Library subview structure)

## 2. Outdated juristic announcements — removal workflow

### Question
How should editors remove an outdated juristic announcement (notice) from a kiosk?

### Answer
Four options ranked best to worst:
1. **Set Expiration Date** on the media doc → kiosk's notice query (`build.mjs:194`) filters `expiresAt > now()` → notice auto-disappears on next build. History preserved.
2. **Toggle `Active = off`** → master switch; hides from rotation AND Building Updates panel. Reversible.
3. **Delete the playlistItem slot** → removes from rotation only; media stays for reuse.
4. **Delete the media doc** → blocked by Sanity's reference integrity until all playlistItems referencing it are removed first.

Key insight: `playlistItem.delete` controls rotation only. `media.isActive` and `media.expiresAt` control all surfaces.

## 3. Kiosk runtime data flow — "Why does adding new media require a deploy?"

### Investigation
User believed the kiosk polled Sanity at runtime. Walked through `vertical-signage.html`, `build.mjs`, deploy/*/index.html, `sw.js`, Netlify configs.

### Findings
The kiosk does **not** poll Sanity in normal operation. Data flow per content type:

| Content type | Baked at build | Fetched at runtime | Poll interval | Cache TTL |
|---|---|---|---|---|
| Playlist (promos+notices) | ✅ `__BAKED__.playlist` | Fallback only when baked empty | None | Until reload |
| Providers + offers | ✅ `__BAKED__.providers` | Fallback only | None | Until reload |
| Notices | ✅ `__BAKED__.notices` | Fallback only | None | Until reload |
| Category Config | ✅ `__BAKED__.categoryConfig` | Fallback only | None | Until reload |
| Video / image files | URLs baked; files via Sanity CDN | SW cache-first | None | Unbounded |
| Page HTML | — | Hourly hard reload from Netlify | 3600s | no-cache |
| Weather / News | — | OpenWeatherMap / newsdata.io | 3600s | localStorage |

### Conclusion
The kiosk loads `window.__BAKED__` exactly once on page load and short-circuits before ever calling Sanity. The hourly `location.reload()` refreshes HTML from Netlify, but Netlify serves whatever `build.mjs` last baked.

**Deploy IS required** for new media to appear. The runtime Sanity fallback is effectively dead code unless the build produces empty arrays.

## 4. Notice expiry mechanism — automated vs manual

### Audit
- `expiresAt` filter exists in **7 places**: `build.mjs:194`, master HTML, 5 deployed kiosk HTML files, and `DirectoryView.tsx`. Filter logic is consistent: `(!defined(expiresAt) || expiresAt > now())`.
- **No automation** to rebuild on expiry: no GitHub Actions, no Sanity webhooks, no Netlify scheduled functions, no cron.
- Editor must manually run `./deploy.ps1` after `expiresAt` passes.

### deploy.ps1 anatomy
```powershell
./deploy.ps1                          # builds + pushes all 5 projects
./deploy.ps1 -Project mahogany-tower  # builds all but pushes only the named project
```
Step 1: `node --env-file=.env build.mjs` — fetches Sanity, writes `../{code}/index.html` for every active project.
Step 2: per-project `git push origin HEAD:master --force`.

- On mount, queried existing slots for the doc's target projects
- Label flipped to "Remove from Playlist" (red tone) when slots existed

- Verified bundle shipped correctly (strings present in `sanity-DV2oR2-_.js`).
- Identified the silent `catch` in `refreshSlotState` swallowing errors → state stays `existingSlots=[]` → label stays "Add to Playlist".

### Redesign (final)

1. **New field `removeFromPlaylistOnPublish`** on `media.ts` — boolean, default false, mutually exclusive with `addToPlaylistOnPublish` at publish time.

   - Both flags ticked → blocked, error dialog
   - `addToPlaylistOnPublish` → existing slot-creation loop
   - `removeFromPlaylistOnPublish` → new `deleteSlots()` function (transaction)
3. **`resetFlags()`** patches the published doc to set both flags back to false after slot ops (one-shot triggers).
4. **`addToPlaylistAction.tsx`** reverted to pre-toggle state with a footnote pointing users to the new checkbox for removal.
5. **Descriptions reworded** on `isActive`, `addToPlaylistOnPublish`, and the new `removeFromPlaylistOnPublish` — plain language, explaining the deploy requirement.

## 8. `isActive` vs `playlistItem` — relationship verification

### Question
Does toggling `isActive` on a media doc create or delete playlistItem documents?

### Answer
**No.** `isActive` is a pure filter field. Confirmed by:
- `media.ts` schema has no `onChange` mutation hook
- `sanity.config.ts` has no subscriber/middleware reacting to it
- `mediaPublishAction.tsx` doesn't read `isActive` at all
- No webhook/Firebase function config anywhere in the repo

`isActive` is read only by GROQ filters at build time (`media->isActive == true`). Toggling it changes whether data flows into the baked output on the next rebuild — never touches `playlistItem` docs.

Three code paths create `playlistItem` documents — none of them check `isActive`:
1. `addToPlaylistAction.tsx` (manual button)
2. `mediaPublishAction.tsx` (when `addToPlaylistOnPublish === true`)
3. `initPlaylistAction.ts` (Initialize Playlist on Project doc)

### Methodology
- GROQ-fetched 3 sample published docs per type
- Extracted schema-declared field names via regex on each schema file
- Computed orphans = data_keys − schema_keys − system_keys
- Cross-referenced UI components for read/write evidence

| ledger | (clean) |

### Methodology corrections
1. **`_system` is NOT an orphan** — defined in `@sanity/media-library-types/lib/index.d.ts:2711` as a Sanity SDK system field with shape `{ base: { id, rev } }`. Safe to ignore.
