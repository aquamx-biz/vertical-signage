Sanity Studio session covering four discrete bug fixes / features: a read-only-document patch crash on the Payment edit view, a new MediaUsageSummary field on Media, a missing-link bug on rent payments created from Rent Space, and a Rent Expense GL code correction.

---

## 1. "Attempted to patch a read-only document" on Payment edit

### Question
User opened a Payment doc edit view and got an "Unhandled Runtime Error: Attempted to patch a read-only document" — twice, at field **2.6 Gross Amount** and field **2.7 W/H Tax**. Sample doc: `03D9pFHqOUR8vo0M6aeG1u`.

### Cause
Two custom inputs auto-fill values from inside a `useEffect` but never check whether the form is currently editable. When Studio renders the form in a read-only state, the effect still fires `onChange(set(...))`, which throws.

The form lands in read-only state in several situations — any of these triggers the crash on this specific doc:

1. **Schema-level `readOnly` hit** — field 2.6 has `readOnly: ({document}) => document?.approvalStatus === 'approved'` (payment.ts:548). After approval the field is read-only but the effect still tries to set it.
2. **Viewing a release / historical version / published perspective** without a draft — Sanity's Content Releases / version perspectives render the doc as read-only.
3. **Missing write permission** for this dataset/role.
4. **Inspector / history pane open** showing a snapshot.

### Fix
Guard both `useEffect` hooks with an early `if (props.readOnly) return` and include `props.readOnly` in the dependency arrays so they re-run if the read-only state changes.

**Files**: `components/AutoPaidAmountInput.tsx`, `components/AutoWhtAmountInput.tsx`

**Commit**: `d617787` — _fix: guard Auto{PaidAmount,WhtAmount}Input onChange against read-only forms_

## 2. MediaUsageSummary — "Used in Playlists" on Media docs

### Question
User asked: from a Media Library doc, can we add a URL link to the Playlist(s) it was added to?

### Answer
Yes, and Sanity's built-in incoming-references tab doesn't solve it. The link chain is `Playlist (view) → playlistItem → media`. The built-in references view on a media doc shows `playlistItem` slot intermediaries, not the parent playlist a user actually navigates to. There's also no `playlist` document type — "Playlist" is a sub-view rendered on a `project` doc at structure path `digital-signage;playlist;<projectId>`.

### Fix
New custom-input field on `media`:

```
playlistUsage   (string, readOnly, custom input: MediaUsageSummary)
```

The component:
- GROQ-queries `*[_type == "playlistItem" && media._ref == $mediaId]` plus joined project info.
- Groups by project, shows badges per slot (`#order`, "off" when disabled).
- Renders an **"Open Playlist"** button per project group that uses `useRouter().navigateUrl({ path: '/structure/digital-signage;playlist;<projectId>' })` for client-side nav to that project's Playlist view.
- Header chips: project count, active-slot count, disabled-slot count (when > 0).
- Empty state: `"Not added to any playlist yet."` — keeps the form quiet for unused media.

**Files**: `components/MediaUsageSummary.tsx` (new), `schemas/media.ts` (import + field after `removeFromPlaylistOnPublish`)

**Commit**: `8ebe09a` — _feat: MediaUsageSummary read-only field on media_

## 3. Rent payments from Rent Space — missing `linkedRentContract`

### Question
User opened a payment doc created via "Record Rent Payment" from a Rent Space billing period (sample: `r9Y99BbHqqRRyRh3D7cn98`, `PMT-2605-002`). Field **1.4 Rental Contract** was empty — why didn't it carry over from the Rent Space?

### Cause
`PeriodPaymentButton.handleCreate` built the new payment payload with `paymentMode`, `paymentStatus`, `paymentAmount`, `currency`, and `vendor` — but **never set `linkedRentContract`**. The bidirectional link `contract.billingPeriods[].linkedPayment ↔ payment.linkedRentContract` only existed in one direction.

### Fix
Add the contract ref to the payment payload at creation time:

```ts
linkedRentContract: { _type: 'reference', _ref: contractId }
```

**File**: `components/PeriodPaymentButton.tsx` (lines 65-72)

Note: this is a creation-time fix only. Existing payments missing the link must be repaired manually via the picker (the field has `disableNew: true` — so use the search to pick the source Rent Space).

## 4. Rent Expense GL code — `512100` → `513100`

### Question
User reviewed the Payment doc's field **4.1 GL Account** and questioned why the auto-fill default for Rent Payment was `512100` instead of `513100` (the correct Rent Expense account in this Chart of Accounts).

### Fix
Two literal swaps in `AutoGlAccountPaymentInput.tsx`:

```ts
// before
const RENT_GL_QUERY = `*[_type == "accountCode" && code == "512100" ...]`
// after
const RENT_GL_QUERY = `*[_type == "accountCode" && code == "513100" ...]`
```

And the matching not-found hint text on line ~206:
```
"513100 Rent Expense account not found — check Chart of Accounts"
```

**File**: `components/AutoGlAccountPaymentInput.tsx`

**Combined commit** (Sections 3 + 4): `4d5d928` — _fix: rent payments created from Rent Space now carry linkedRentContract reference, and Rent GL defaults to 513100 Rent Expense_
